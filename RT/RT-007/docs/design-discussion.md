# RT-007 设计理念与讨论备忘录 (Design Discussion Log)

**时间**: 2026-04-09
**参与方**: Evan & Antigravity (ARC Reactor 维护组)

## 1. 从“死存储”到“活维基”的质变诉求
前期 RT-006 我们通过 Karpathy LLM Wiki 架构实现了知识的持久化（Compilation over Retrieval）。但在实际生产中发现，如果用户在下次提问时还需要手动给 Agent 喂文件路径，知识库就失去了“外脑”的意义。我们提出了**随用随取的自动潜意识投喂**机制。

## 2. 传统 RAG 与 ARC Wiki 的核心鸿沟
讨论中我们对比了传统向量化 RAG 与当前 Markdown 格式的劣势与优势：
| 机制 | 痛点/劣势 | 我们的解法 (ARC Context Hook) |
|---|---|---|
| **传统 RAG** | 把文本切碎成无语境的几十 Cunks，经常牛头不对马嘴。 | 我们提取出的是一张张**完整的个人心血卡片 (`entities/xxx.md`)**，保留了完整的论述生命线。一次性抛给顶级大模型，推理降维打击。 |
| **调用麻烦** | 需要建设重度的索引库、向量计算环境。 | 一套基于纯文本匹配和正则提取的脚本 `context-injector.py` 即可撬动巨大的知识网络。 |

## 3. 三大核心落地方案展望
为支撑“用”的终极目标，我们勾勒出以下形态：
1. **智能预取（挂件）**：通过脚本在提示词喂给大模型前快速拼接，本次 RT-007 直接攻坚此能力。
2. **多域周报**：利用源文件的 `sources/YYYY-MM-DD` 的自然时间线，用 Cron Jobs 主动向管理者提取本周增量的报告总览。
3. **冲突对撞机**：对于新旧观念进行冲突审计（例如过去反对某项技术，今天新摄入卡片又推荐，让代理层自发提报这种冲突）。

> **总结定调**：
> 不要过度依赖网关层（如 OpenClaw 底层），尽可能将知识预取拦截做成“脚本+SKILL 指令”的双层驱动，以此实现最高自由度的分发与插拔！
